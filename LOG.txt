2026-07-13 — Phase 0/1 (project pivot from AXTermPuter, scaffold + KISS codec)
- New sibling project to AXTermPuter (~/Dev/Projects/AxTermPuter). Same
  feature scope (APRS messenger + AX.25 terminal for the UV-Pro), but as a
  Python desktop/laptop app instead of Cardputer firmware, because the
  AXTermPuter investigation this session confirmed the UV-Pro's real
  KISS/TNC data path is classic Bluetooth RFCOMM, which the Cardputer's
  ESP32-S3 (BLE-only radio) cannot reach at all. Full findings carried
  over into this repo's docs/PROTOCOL.md §2. Laptops/tablets (Chuwi
  MiniBook, HiMax10, both Kubuntu) have real classic-BT radios, so this
  project can talk to the radio's actual transport directly.
- Created CLAUDE.md, ROADMAP.md, docs/PROTOCOL.md, LOG.txt.
- Decisions: Python 3.12+; classic BT via stdlib
  `socket.AF_BLUETOOTH`/`BTPROTO_RFCOMM` (no extra dependency, Linux-
  native); urwid for the terminal UI; pytest for tests; PyInstaller +
  GitHub Actions for Linux (primary) and Windows (stretch) executable
  releases. All daily-use devices are Kubuntu — don't let Windows
  compatibility block Linux progress.
- Added uvprotermbt/kiss.py: direct port of AXTermPuter's native-tested
  lib/kiss/kiss_codec.{h,cpp} — encode_frame() and streaming KissDecoder
  with the same design decisions (empty-frame skip, malformed-escape
  resync).
- Added tests/test_kiss.py: ported the same 8 cases from AXTermPuter's
  test/test_kiss/test_kiss_codec.cpp (encode escaping, single/multi-frame
  decode, empty-frame skip, fragmentation at every split point, malformed-
  escape resync, 300-byte payload, deterministic fixed-seed fuzz test).
- VERIFIED: `pytest tests/` PASSED (8/8), via a local .venv.
- NOT done: git push to github.com/chengmania/UVProTermBT (repo has no
  commits yet).
- NEXT: Phase 2 — RfcommKissLink (classic BT RFCOMM connect from Linux,

2026-07-13 — Phase 2 (RFCOMM link) — implemented, channel corrected, still flaky live
- Added uvprotermbt/link.py: RfcommKissLink (socket.AF_BLUETOOTH/
  BTPROTO_RFCOMM), background RX thread + queue, send()/on_receive()/
  poll(), reconnect with backoff. Added scripts/monitor.py as a Phase-2
  hardware smoke test (connect, print raw + KISS-decoded frames).
- Confirmed via `sdptool records <addr>` (not `browse`, which comes back
  empty) from this machine directly: SDP shows RFCOMM channels 2 ("BS
  AOC"), 3 (Voice Gateway/HFP), 4 ("SPP Dev"/Serial Port 0x1101).
  Initially targeted channel 4 as the "proper" SDP-discoverable choice.
- CORRECTION: found the khusmann/benlink project on GitHub — a Python
  library specifically reverse-engineering this exact radio family
  (Benshi: BTech UV-Pro, RadioOddity GA-5WB, Vero VR-N76/N7500, BTech
  GMRS-Pro). It documents the real BLE control service
  (00001100-d102-11e1-9b23-00025b00a5a5 — not the 00000001-ba2a-... one
  we'd found, which is itself a red herring) and a GaiaFrame/Message
  command protocol. Its own RFCOMM example uses channel 1, not 4.
  Cross-checked against our earlier captured bytes: channel 4 traffic
  ends in `...ff 01 00` (a GaiaFrame header), channel 1 traffic starts
  with `c0 00` (raw KISS FEND + command byte, no envelope). Conclusion:
  channel 4 is a red herring (generic serial profile, not the real TNC
  path); channel 1 carries genuine raw KISS pass-through. Changed
  RfcommKissLink.DEFAULT_CHANNEL from 4 to 1. Full writeup in
  docs/PROTOCOL.md §2. Decided NOT to adopt benlink's Message/GaiaFrame
  protocol for our core KISS data path — our own kiss.py + RfcommKissLink
  on channel 1 is simpler and sufficient; benlink stays a good reference
  for future radio-control features (battery, channel switching).
- VERIFIED live from Linux (twice): a bare RFCOMM connect to channel 1
  succeeds cleanly after a fresh `bluetoothctl pair`. Neither test window
  happened to have live RF traffic, so we haven't yet decoded a real
  AX.25 frame end-to-end through this project — transport-level
  connection is confirmed, full data-path confirmation is still pending.
- NOT resolved: classic BT connection reliability. Spent a large amount
  of this session on live hardware debugging — saw clean connects,
  TimeoutError, ConnectionRefusedError, and BlueZ-level
  br-connection-refused (that last one specifically from an asymmetric
  bond: our side had removed its copy of the link key while the radio
  still remembered pairing with us — symmetric removal on both sides
  fixed that particular case). The radio's own "paired device" indicator
  was observed appearing and disappearing within a few seconds,
  independent of anything we did. A full power cycle (battery pull)
  restored BLE connectivity once when it had gone flaky. Holding a BLE
  connection open at the same time does NOT stabilize classic RFCOMM —
  confirmed independent. No access to the radio's own firmware logs, so
  further root-causing from software has hit its limit for tonight.
- NEXT: get a genuine live end-to-end KISS/AX.25 decode through
  RfcommKissLink on channel 1 (needs to coincide with actual RF traffic
  during a working connection window). If classic-BT flakiness keeps
  blocking progress, consider a dedicated session focused purely on
  characterizing the radio's connection-window timing before continuing
  with Phase 3 (AX.25 parsing) against real traffic — though Phase 3's
  parsing logic itself doesn't require live hardware and could proceed
  independently using known-good captured byte sequences as test fixtures.

2026-07-13 — Phase 2 (RFCOMM link) — root cause found and fixed
- User found two independent real-world writeups documenting a working
  classic-BT KISS-TNC setup for this exact radio family (VR-N76/BTech
  UV-Pro) on Linux: a HamRadioTech article (DC6AP, Sep 2025, "VR-N76
  (UV-PRO) with KISS-TNC under Linux") and the
  islandmagic/kiss-tnc-test Raspberry Pi guide. Both independently
  confirm RFCOMM channel 1 (matching our own khusmann/benlink-informed
  finding), and both call out disabling BlueZ's Bluetooth headset/
  Hands-Free profile as a required fix we'd never tried.
- Confirmed the root cause directly on this machine via
  `journalctl -u bluetooth`: repeated, exactly-timed log lines
  throughout every flaky session tonight —
  `bluetoothd[1256]: src/profile.c:ext_connect() Hands-Free unit failed
  connect to 38:D2:00:01:38:8F: Connection refused (111)`. BlueZ's
  `hfp-hf` (Hands-Free unit/client role) plugin auto-connects to the
  radio's advertised Handsfree Audio Gateway service (SDP `Voice
  Gateway`/0x111f, channel 3), contending with our RFCOMM channel-1 KISS
  connection attempts at the BlueZ/kernel level, regardless of which
  client API initiates the connection.
- The two articles' own fix (`/etc/bluetooth/input.conf`,
  `Disable=Headset`) is stale for this system's BlueZ 5.72 — confirmed
  that option doesn't exist in this version's input.conf (HID-only).
  Found the modern equivalent: `bluetoothd --help` documents
  `-P/--noplugin=NAME`; `strings /usr/libexec/bluetooth/bluetoothd`
  confirmed compiled-in plugin names `hfp-hf`/`hfp-ag`/`hsp-hs`/`hsp-ag`.
- APPLIED (user-confirmed, system-wide change): created
  `/etc/systemd/system/bluetooth.service.d/override.conf` disabling only
  `hfp-hf` (the exact role in the failing log line):
  ```ini
  [Service]
  ExecStart=
  ExecStart=/usr/libexec/bluetooth/bluetoothd --noplugin=hfp-hf
  ```
  Applied via `sudo systemctl daemon-reload && sudo systemctl restart
  bluetooth`; verified new bluetoothd process running with the flag.
  A2DP (music/audio playback) endpoints still register normally —
  unaffected. Rollback: remove the override.conf, daemon-reload, restart.
- Updated pairing procedure to include `trust <MAC>` after `pair <MAC>`
  (both external articles use it, we never had) — documented in
  docs/PROTOCOL.md §2 and scripts/monitor.py's docstring.
- Architecture decision: keep RfcommKissLink's raw
  socket.AF_BLUETOOTH approach rather than switching to `rfcomm bind` +
  pyserial (which both external guides use) — the root cause was at the
  BlueZ/kernel level below either client API, so their reliability is
  attributable to the headset fix, not to `rfcomm bind` itself. No
  proven benefit to switching; would add a sudo-gated system device
  node, a new pyserial dependency, and a duplicate reconnect lifecycle.
  Documented as a fallback option if raw sockets are still unreliable
  after this fix.
- Polished uvprotermbt/link.py: differentiated ConnectionRefusedError/
  TimeoutError/generic OSError in _run()'s logging for actionable
  diagnostics (still same backoff behavior, no control-flow change);
  added a threading.Lock around _sock/_connected mutation in send()/
  _handle_disconnect() (defensive cleanup for a previously-benign race
  between the main thread's send() and the RX thread's own disconnect
  handling); send() failures are now logged instead of silent.
- VERIFIED: `pytest tests/` still 8/8 passing after the link.py changes.
- Documented direwolf's TX limitation with this radio (HamRadioTech:
  direwolf RX works, TX silently fails, cause unknown; recommends
  TFKISS) — noted for later Phase 4+ on-air TX testing, not installing
  new software this session since our current hardware test is RX-only.
- NEXT: live hardware re-test with the fix in place — re-pair with
  `remove` + `trust`, run scripts/monitor.py, confirm a connection that
  stays up (not flapping), and catch a real live KISS/AX.25 decode
  during actual RF traffic to finally close out the Phase 2 hardware
  test.
